# grok-regkit

把 **Grok 账号注册** 做成可跑的工具包：自动出号，拿到 **SSO**，并可导出 **OIDC（CPA `xai-*.json`）** 给 free **Grok 4.5**。自带 Web 控制台、GUI 与 CLI。

**当前定位：注册机 + 凭证导出** · 双模式（浏览器 / 混合协议）· CPA 协议优先 mint

> **SSO ≠ OIDC。** 只有 SSO 不能当 Build 4.5 凭证。详见 [sso-cpa/](./sso-cpa/)。

---

## 亮点

| 能力 | 说明 |
|------|------|
| **双注册模式** | `browser` 全浏览器求稳 · `hybrid` 协议 RPC + 短浏览器采 token 求快 |
| **真浏览器过盾** | Chromium 采集 Turnstile / castle，贴近真实注册页环境 |
| **SSO 落盘** | `邮箱----密码----sso` 写入 `accounts_*.txt`，可选写入号池 |
| **CPA / OIDC 一条龙** | 注册成功后自动 mint → `cpa_auths/xai-*.json` → CLIProxyAPI 热加载 → **grok-4.5** |
| **协议优先 mint** | SSO HTTP device-flow 优先；失败可浏览器回退；可配间隔防 429 |
| **wrapper → session** | Hybrid 常见 wrapper SSO 可 materialize 成可用 session |
| **多入口** | Web 控制台 · GUI · CLI · backfill 脚本 |
| **凭证边界清晰** | 文档与管线区分号池（SSO）与 4.5（OIDC），减少混用 Key |

## 架构

```text
临时邮箱 + 代理
        │
        ▼
  grok-regkit  (browser | hybrid)
        │
        ▼
     SSO cookie
        │
   ┌────┴────┐
   ▼         ▼
 号池      CPA mint (OIDC)
 Web 模型  xai-*.json → CLIProxyAPI → grok-4.5
```

| 阶段 | 内容 |
|------|------|
| 注册 | `accounts.x.ai` 邮箱验证码 + 资料提交 + 过盾 |
| SSO | set-cookie / materialize → 会话 JWT |
| OIDC | device-flow 或浏览器确认 → 本地 auth JSON（`type=xai` 管线） |
| 后处理 | 可选 NSFW、号池入池、CPA 探活 |

## 注册模式

| 模式 | 配置 | 特点 |
|------|------|------|
| **browser** | `"register_mode": "browser"` | 全 Chromium，过盾与页面状态完整，适合日常 |
| **hybrid** | `"register_mode": "hybrid"` | 短浏览器采 token + 协议发业务请求，通常更快；SSO 需注意 session 化 |

两者成功后共用后处理：落盘 →（可选）入池 →（可选）CPA mint。

### 混合模式快多少？

没有固定「快 N 倍」——单号总耗时主要卡在 **过盾、收验证码、代理质量**，和模式无关的部分两边差不多。

hybrid 省的是 **整页点选、填表、等 UI 跳转** 的那段：

| 阶段 | browser | hybrid |
|------|---------|--------|
| 打开注册页 / 过 CF 相关 | 浏览器全程 | 浏览器仍要开（采 token） |
| 提交邮箱、验证码、建号 | 页面操作，步骤多、等待多 | 协议 RPC / Server Action，更短 |
| Turnstile / castle | 页面内完成 | 浏览器短采，业务侧再带上 token |
| 浏览器占用 | 整号生命周期都挂着 | 窗口更短，采完即可收 |

经验量级（环境正常、一次成功时）：

- **全浏览器**：单号常见更长，UI 步骤多，内存占用高  
- **混合**：业务段明显缩短；总耗时往往能少一截（很多情况下大约 **省 20%～40% 墙钟时间** 量级，过盾慢时差距会缩小）  
- **过盾/邮箱很慢时**：两种模式都会被拖住，hybrid 的优势主要变成「少点一点页面、少占一点浏览器」

后处理 CPA mint 两边同一套，不计入「注册模式谁更快」。

### 为什么不做默认「纯协议」？

纯协议 = 注册请求全 HTTP，Turnstile 用 **打码平台**（如 YesCaptcha createTask）换 token，而不是本地开 Chrome 采。

我们 **默认不走这条**，主要因为：

1. **打码有成本**  
   每号、有时每步都要买 token，批量出号费用会叠上去；本项目优先 **本地 Chromium 过盾，不强制付打码费**。

2. **还要绑第三方账号与额度**  
   打码 Key、余额、排队、服务商故障都会变成新单点；混合/浏览器路径只依赖你自己的 Chrome + 代理。

3. **CF / 风控不只有 Turnstile 一关**  
   纯 HTTP 靠 TLS 指纹 + 打码 token；遇到硬拦截、要 `cf_clearance` 或页面环境时，浏览器路径往往更扛打。纯协议在「打码成功」时可以很快，但失败时整批更容易一起跪。

4. **混合已经吃掉「协议快」的大半**  
   业务用协议，盾用浏览器采 —— **又快一点，又不必为每号付打码**。这是成本和成功率之间的折中。

5. **纯协议可作以后可选，而不是唯一默认**  
   若你自备打码、追求极限并发与更轻服务器，可以再加 `protocol` / YesCaptcha 插件；默认路径仍保持「零打码费用、混合加速」。

一句话：**混合 = 用协议省时间，用本地浏览器省打码钱；纯协议 = 可能更快更轻，但每号打码成本和第三方依赖是硬门槛。**

## 凭证说明

| 凭证 | 形态 | 用途 |
|------|------|------|
| **SSO** | Cookie `sso` | 号池 / Web 反代模型（如 4.20、4.3） |
| **OIDC** | `access_token` + `refresh_token` | CLIProxyAPI → **grok-4.5** |

```text
SSO  ──► 号池 ──► Web 模型
SSO  ──► mint ──► xai-*.json ──► CPA ──► 4.5
```

调用时不要混用：号池 Base + 号池 Key ≠ CPA Base + CPA Key。

## 快速开始

```bash
cd grok-regkit
python -m venv .venv
# Windows
.venv\Scripts\activate
pip install -r requirements.txt
cp config.example.json config.json
# 编辑：邮箱 API、代理、register_mode、CPA 开关
```

### Web（推荐）

```bash
# Linux 无桌面时: export DISPLAY=:99  （Xvfb）
uvicorn web.server:app --host 127.0.0.1 --port 8092 --workers 1
```

浏览器打开：`http://127.0.0.1:8092`

### GUI / CLI

```bash
python grok_register_ttk.py
python grok_register_ttk.py --cli
```

### 常用配置

| 字段 | 说明 |
|------|------|
| `register_mode` | `browser` \| `hybrid` |
| `email_provider` | `cloudflare` \| `duckmail` \| `yyds` |
| `cloudflare_*` / `duckmail_api_key` / `yyds_api_key` | 对应邮箱密钥（见下） |
| `proxy` / `proxy_mode` | 注册代理 |
| `cpa_export_enabled` | 是否注册后 mint OIDC |
| `cpa_prefer_protocol` | 协议 mint 优先 |
| `cpa_auth_dir` | 默认 `./cpa_auths` |
| `cpa_mint_gap_sec` | mint 间隔，防限流 |

完整字段见 [sso-cpa/03-config.md](./sso-cpa/03-config.md) 与 `config.example.json`。

### 邮箱后端

| `email_provider` | 配置项 | 说明 |
|------------------|--------|------|
| `cloudflare` | `cloudflare_api_base`，可选 `cloudflare_api_key` / `cloudflare_auth_mode` | Cloudflare Temp Email 兼容 Worker |
| `duckmail` | `duckmail_api_key` | DuckMail |
| `yyds` | `yyds_api_key`，可选 `yyds_jwt` | YYDS Mail（建号 + 轮询验证码） |

browser / hybrid 共用同一套邮箱逻辑。Web 控制台有对应三个页签。

更细的说明、以及对照 **chatgpt2api** 注册邮箱栈（GPTMail / Outlook token / MoeMail / 域名黑名单等）的适配参考，见：

→ [mail-providers.md](./mail-providers.md)
